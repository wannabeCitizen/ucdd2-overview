Web Servers in Javascript

What goes into a web service?

Primary Concepts:

  • Client = The browser rendering/receiving the page
  • Server = The computer hosting the website and data
  • HTTP Protocol = How messages are sent between different agents (i.e., clients and servers) on the web

HTTP Protocol

  • Communication protocol (most common HTTP 1.1, but HTTP 2.0 is being hacked on now)
  • Uses headers to provide metadata about the request (e.g, what IP it's for, user-agent, origin-domain, data types, etc)
  • Uses a body to provide data or DOM content
  • Clients and Servers are pre-programmed to do a lot of things that prevent hacking (probably why you banged your head a bunch)

Client

  • Renders a DOM (Document object model), which is essentially a tree where each vertex corresponds to something rendered on the page
  • DOM gets build as HTML with CSS marking it up (ie., styling it)
  • Browsers have compilers (most use V8) and run single-threaded, asynchronous processes that run the compiled JavaScript and consume the DOM (i.e., read the DOM, do computations, update the DOM)
  • Client's use the IP address of the computer running them

Server

  • Accepts requests through a port that is being listened to by a webserver (for websites, port 80)
  • Requests have their headers considered then their body, and the webserver then decides how to respond given this information (APIs often respond with raw data, but websites with a DOM)
  • Web servers mediate the relationship between the client, any database, and a set of files that represent the different pages on the website

So JavaScript...

It's now used "full stack":

  • Front-end JS expects a DOM (i.e., document) to consume, this was originally added to the web (ECMAScript) in order to allow the web to be dynamic
  • Server-side JS began with Node where the V8 compiler was written into an interpreter than runs on File Systems (not DOMs!)
  • Databases (No-SQL) have begun adopting the JS Object (ie, {key: value} ) as a means of storage also

Tools!

Node and NPM

  • Node = JavaScript running on a file system
  • NPM = Online library of downloadable packages, making it easy to add them to an application you're building
  • It's common to use dependency managers (eg, Bower or RequireJS) and build systems (eg, gulp, wintersmith, or grunt) so that you can write your application's logic in separate modules without worrying about scope and dependencies, but then you run your build script to create your real website

Express

  • Most common web server used for Node
  • Gives you all the tools you need to listen to a port, handle a request, and generate a response
  • Will not, by default, merely expose all files (security issue!), thus you must define what files are exposed, whether a database exists, and what URLs anticipated
  • Also can help you do sessioning and authentication; though you'll often use some other tools together with Express

Jade and Wintersmith

  • Jade is nothing more than shorthand HTML. It also helps in modularizing your file since we commonly use the same headers, footers, and navbars across views
  • BUT! Jade is NOT HTML, and thus it requires a build engine to be converted
  • Wintersmith is a build engine for making static websites, thus it is often used with Jade
  • Wintersmith provides in separating out static views, modules, plugins, etc so you can patch them together into a static page when building

Mocha

  • This is a testing framework for JavaScript
  • Often used in tandem with a build engine so you can set a task that runs all your functionality tests before building
  • Another testing framework is Jasmine, and almost anyone you talk to will use one of these two.

MongoDB

  • No-SQL Database = instead of table with rows and columns, we have documents of keys and values
  • Need to establish a connection as Mongo is read from a socket (cf. TCP/IP protocol to understand sockets)
  • Uses a JavaScript-like syntax for querying, making it very convenient for Node applications
  • There are many libraries (eg, Mongoose) that can help you handle your Mongo CRUD operations cleanly

JavaScript Libraries

  • JQuery is a library with common utilities to help manipulate the DOM with JavaScript
  • LoDash or Underscore provides common operations used working with data (ie, the model)
  • For Client, load these in a script tag, then any subsequent JS can expect these function in scope
  • For Server, must require them as a module then can use their functionality
  • Be Careful! Always read source. For instance, JQuery's .get() function caused many people headaches because it sets headers in an assumptive manner

NOTE: JS Libraries are different than CSS Libaries (like Bootstrap CSS)

However many web frameworks often offer both..

MVC Paradigm

  • MVC = Model, View, Controller -- it is a common approach for thinking about web applications.
  • Model = The data that is relevant to a particular state of a web application
  • View = A particular rendering of data and web components. This can be controlled through static paths (URLs) or with a dynamic framework like React or Angular
  • Controller = The logical backbone of your application that works to ensure the model is transformed appropriately, events are triggered on time, and server-client communications are done correctly.

React JS and other frameworks

  • Each of these "web frameworks" provides a way of thinking about MVC/Web App Design.
  • React is meant to help your separate out components of your web page that have distinct logic, often controlled by "events" or "streams"
  • Thinking in "React" is mostly deciding what parts of your view need what data then determining hierarchies of components that chan share data.
  • React is mostly a library that helps you give your view components logic about how they render and change.
